Problem Statement

Current Challenges

  • Traditional IoT systems rely on WiFi-only communication — limited range & high power consumption
  • No standardized method for dynamic sensor configuration at firmware level
  • Existing systems require manual firmware rewrites for each node
  • Tightly coupled hardware-software designs prevent code reuse

Critical Gap

There is no lightweight, modular, and JSON-configurable multi-node IoT gateway framework that combines RS485 wired reliability with MQTT cloud connectivity — designed for microcontrollers.

RS485 Communication Flow

Half-Duplex Protocol with Direction Control

1
Master sets DE/RE pin HIGH → TX mode
→
2
Master sends request frame with Node ID
→
3
Master sets DE/RE pin LOW → RX mode
→
4
Node responds with sensor data frame
RS485Handler.cpp — Direction Control 
void RS485Handler::sendData(const String& data) {
    gpio->digitalWrite(dirPin, HIGH);  // TX mode
    delayMicroseconds(100);
    serial->println(data);
    serial->flush();
    gpio->digitalWrite(dirPin, LOW);   // Back to RX
}

String RS485Handler::receiveData(unsigned long timeout) {
    gpio->digitalWrite(dirPin, LOW);   // RX mode
    unsigned long start = time->millis();
    while (time->millis() - start < timeout) {
        if (serial->available()) {
            return serial->readStringUntil('\n');
        }
    }
    return "";
}

JSON Configuration Model

Dynamic Firmware Configuration

Instead of hardcoding sensor logic, each node reads its behavior from a compile-time JSON config stored in configJSON.h. This enables:

  • Swap sensors without rewriting main.cpp
  • Define pin mappings, polling rates, and node IDs
  • Auto-generate config from Web UI
configJSON.h — Example Node Configuration

  "nodeId": 1,
  "role": "node",

    "dataPin": 15,
    "pollIntervalMs": 2000
  },
  "rs485": {
    "uartId": 0,
    "txPin": 0,
    "rxPin": 1,
    "dirPin": 2,
    "baudRate": 9600
  }
})";

WiFi + MQTT Integration

📶 ESP8266 WiFi Bridge

  • ESP-01 module connected to Master Pico via UART1
  • Controlled using AT commands
  • Handles TCP connection to MQTT broker
  • Master Pico sends formatted payloads

📨 MQTT Protocol

  • Lightweight pub-sub messaging (ideal for MCUs)
  • Topics: iot/node/{id}/data
  • QoS Level 0 for speed
  • Mosquitto broker on laptop
MQTTBridge.cpp — Publishing Sensor Data 
void MQTTBridge::publish(uint8_t nodeId, const String& payload) {
    String topic = "iot/node/" + String(nodeId) + "/data";
    String mqttPacket = buildPublishPacket(topic, payload);
    
    espSerial->println("AT+CIPSEND=" + String(mqttPacket.length()));
    delay(100);
    espSerial->print(mqttPacket);
    
    // Wait for ESP8266 ACK
    waitForResponse("SEND OK", 5000);
}

Demo Workflow

1

Configure via Web UI

Open firmware generator → select sensor type, pins, node ID → download configJSON.h

2

Flash Firmware

Place configJSON.h in project → compile with Arduino IDE → flash to Pico via USB

3

Connect Hardware

Wire sensor to Node Pico → connect RS485 bus → attach ESP8266 to Master Pico

4

Power Up System

All nodes boot → Master polls each node in round-robin → nodes respond with sensor data

5

View on Dashboard

Open MQTT client / dashboard on laptop → see real-time sensor readings from all nodes

Advantages

🔌

Wired Reliability

RS485 provides noise-immune communication up to 1200m — no WiFi dependency for node-to-master link

💡

Ultra Low Cost

Entire system under $30 — Pico ($4) + MAX485 ($0.5) + sensors ($2-5 each)

🧩

Plug & Play Sensors

JSON config model lets you swap sensors by changing a config file — no firmware rewrite

🏗️

Clean Architecture

HAL + dependency injection = testable, portable, maintainable codebase

📈

Scalable

RS485 bus supports up to 32 nodes — add new nodes simply by flashing new Picos

🌐

Cloud Ready

MQTT integration enables dashboards, alerts, logging, and cloud analytics pipelines
